Evaluation Criteria
Submissions are assessed across three dimensions:
ID Criterion Description
P1CorrectnessAccurate and precise answers for the given queries
P2Explanation QualityClear, coherent natural-language explanations that
justify each answer
P3Reasoning DepthSupporting evidence such as FOL derivations,
Chain-of-Thought steps, or structured proofs
Evaluation process:
1 Phase 1 (Selection):Submissions are scored automatically against ground-truth answers,
then reviewed by the organizing committee for explanation quality. Detailed feedback is
sent back to teams.
2 Phase 2 (Selection):Same evaluation pipeline as Phase 1 (auto correctness +
committee review of explanations) after teams refine their models.
3 Final Round (Public Test Day):Top 10 teams run their systems live on unseen queries.
The Challenge Chairs evaluate answers, explanations, and reasoning depth in real time.
Final score = weighted combination of P1, P2, P3. Specific weights will be published with the
official dataset release.

Type 1: Problem Definition
Objective
Build a system that answers questions aboutuniversity academic regulations(grading
policies, enrollment rules, scholarship criteria) and provideslogical explanationsgrounded in
First-Order Logic (FOL).
Why is this challenging?
Regulations contain complex conditional logic (if-then rules, exceptions, nested conditions)
Answers must be provably correct, not just plausible
Explanations must trace back to specific regulation clauses
Requires bridging natural language understanding with formal logic

Type 1: Dataset Format
Statistics
411 recordscontaining808 questionsin total.
Question types:358 Multiple Choice, 216 Yes/No/Uncertain, 234 Open-ended.
Coverage:99% of records include FOL premises; 100% include human-written explanations.
Average per record:∼10.8 NL premises,∼1.97 questions.
Each record includes:
Field Description
Natural-language premises The regulation text describing rules and conditions
FOL annotations First-Order Logic representation of the premises
Questions One or more questions about the regulation
Ground-truth answers Correct answers (Yes/No/Uncertain or specific values)
Human-written explanations Step-by-step reasoning explaining why the answer is correct

Type 2: Problem Definition
Objective
Build a system that solvestext-based physics problemsfocusing on electric circuits and
electrostatics, and providesstep-by-step Chain-of-Thought reasoningfor each solution.
Why is this challenging?
Problems require multi-step numerical computation
Must correctly identify which physics formulas to apply
Intermediate steps must be shown and logically connected
Final answers must include correct units
Open-source LLMs (8B parameters or fewer) often struggle with precise arithmetic

General Architecture
A typical system for EXACT 2026 may combine:
1 Natural Language Understanding:Parse the input query, identify question type (Type
1 or Type 2), extract key entities and constraints
2 Reasoning Engine:Apply logic-based or computation-based reasoning
Type 1: Symbolic reasoning (e.g., Z3 Solver), FOL theorem proving
Type 2: Equation solving, symbolic math, step-by-step computation
3 Explanation Generation:Produce natural-language explanations aligned with the formal
reasoning steps
4 Answer Formatting:Structure the output in the required API format
Note: This is one possible architecture. Any approach that produces accurate, explainable
results is welcome.

Approaches for Type 1 (Logic-Based)
Symbolic Reasoning
Convert NL premises to FOL using an
LLM
Feed FOL into a solver (Z3, Prover9,
custom engine)
Use solver output to generate
explanations
Pro:Provably correct answers
Con:NL-to-FOL conversion is
error-prone
Neurosymbolic Hybrid
Use LLM for understanding + symbolic
engine for reasoning
LLM generates candidate FOL, solver
verifies
Iterative refinement: if solver rejects,
LLM re-generates
Pro:Combines flexibility with rigor
Con:More complex pipeline
Pure LLM with Prompting
Use Chain-of-Thought, few-shot examples, and self-consistency checks. Simpler to implement
but may lack formal guarantees.

Approaches for Type 2 (Physics)
LLM + Code Execution
LLM generates Python/SymPy code for
computation
Execute code to get precise numerical
answers
Use LLM to narrate the steps
Pro:Precise arithmetic
Con:Code generation errors
Retrieval-Augmented Generation
Build a knowledge base of physics
formulas and solved examples
Retrieve similar problems as few-shot
examples
LLM solves with retrieved context
Pro:Leverages existing solutions
Con:Limited by retrieval quality
Practical Tips
Fine-tune on the training data to adapt the LLM to the specific question format
Use self-verification: have the model check its own answer by substituting back
Ensemble multiple approaches and select the most consistent answer
